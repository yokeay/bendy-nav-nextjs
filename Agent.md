1、版本全局维护一个版本，留好配置文件的参数；版本分为三个板块，大中小，三个版本，大版本每次大更新我会跟你说，中版本每次发布我会跟你讲，小版本就是每次提交之后维护一下；
2、创建maintain.md,记录好每一次版本迭代的记录，做了哪些修改，如何修改，怎么做的，记录好每一次版本的功能；
3、创建plan.md，记录好当前项目的已经规划好的所有迭代任务，然后根据迭代任务进行操作，做完的任务记得标记为已做完，按照plan推进项目；
4、webUI样式：不允许改，就按当前老项目来；
5、每次迭代完，自动执行lint扫描，执行build，若不通过则修改，通过后自动commit代码，然后push至远程origin仓库；
6、分支策略：main 分支为生产分支，dev 为开发分支；每个迭代任务从 dev 拉 feat/任务名 分支开发，完成后合并回 dev；中版本发布时 dev 合并到 main 并打 tag，开发完要由我验收后在批准合入main分支。
7、测试规范：核心安全模块（TOTP 生成验证、Token 签发吊销、会话管理）必须编写单元测试，覆盖率不低于 80%；每次迭代新增功能必须附带对应测试；
8、安全规范：所有密钥和敏感配置走 .env 文件，绝不硬编码，.env 必须在 .gitignore 中；API 接口全部要求认证，敏感操作（踢人、吊销 Token）需二次验证；TOTP 密钥存储必须加密；所有用户输入必须校验和转义，防 XSS/SQL 注入；HTTP 接口强制 HTTPS，所有依赖外部的配置，如数据库地址，redis地址，s3存储等信息均在.env中配置，创建，.env.example文件示例配置；
9、错误处理：前端统一封装 API 请求层，所有接口错误走统一的 toast/notification 提示；后端统一错误码规范（如 4 位数字：1001 Token 过期、1002 权限不足等），返回格式统一为 {code, message, data}；日志使用结构化格式，关键操作（登录、踢人、Token 操作）必须留审计日志，项目如有后台管理，需要留有日志审计入口；
10、DB 迁移策略 — 数据库结构变更用迁移文件管理（如 Prisma Migrate、golang-migrate），不要手动改表。
11、Docker 容器化 — 维护 Dockerfile 和 docker-compose.yml，一键启动开发环境。
12.、i18n 国际化 — 从第一天就用 i18next 或 vue-i18n 把文案抽出来，降低语言切换成本。
13、依赖安全扫描 — 每次构建前跑 npm audit 或 pnpm audit，有高危漏洞不允许发布。
14、Commit 规范：使用 Conventional Commits 格式，如 feat: 添加 TOTP 绑定页面、fix: 修复 Token 刷新逻辑、chore: 升级依赖；自动 commit 时格式为 chore(release): v{版本号} 迭代完成。
15、备份与灾备 — 数据库定时备份策略，此策略需要根据配置文件来，如果配置文件开启灾难备份与恢复，则系统在初始化后创建对应的计划，注意TOTP 密钥有独立的加密备份方案，执行之前如果需要额外配置，一定要在readme文档，maintain文档中做好充分记录，以及在环境配置env.example中做好充分记录，默认是不开启的。
16、业务前缀，数据库，数据表，redis缓存等，都基于业务前缀做区分，每个不同业务前缀一定不同，使用的时候一定要带业务前缀，业务前缀一般配置在readme文档中，自行去看：当前业务前缀。
